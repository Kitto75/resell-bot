# ربات مدیریت ریسلر تلگرام برای Marzban

این پروژه یک ربات تلگرام برای مدیریت ریسلرهای Marzban است. رابط کاربری ربات، دکمه‌ها، پیام‌های خطا، تاییدها، پنل مدیر و پنل ریسلر فارسی هستند؛ فقط مقدارهای فنی دریافتی از API مرزبان ممکن است انگلیسی باشند.

## امکانات

- مدیریت ریسلرها توسط مدیر: افزودن، ویرایش، فعال/غیرفعال‌سازی و مشاهده لیست کامل ریسلرها.
- نمایش موجودی و قیمت‌ها با واحد تومان، بدون اعشار و با جداکننده هزارگان؛ نمونه: `100,000 تومان`.
- ساخت اکانت مرزبان با وضعیت `on_hold` و ارسال مقدار معتبر `on_hold_expire_duration`.
- ثبت مالکیت اکانت در فیلد note مرزبان با قالب ساده `belongs to reseller-name`.
- جلوگیری سمت سرور از مشاهده یا تمدید اکانت‌هایی که متعلق به ریسلر نیستند.
- تمدید اکانت با تایید دکمه‌ای و نمایش خلاصه فارسی، حجم به گیگابایت، زمان به روز/ساعت و آخرین برنامه / User-Agent.
- درخواست شارژ توسط ریسلر و تایید/رد دکمه‌ای توسط مدیر با ثبت تراکنش موجودی.
- گزارش‌های فارسی با تاریخ شمسی؛ منطقه زمانی Asia/Tehran به‌صورت ضمنی استفاده می‌شود و در متن گزارش نمایش داده نمی‌شود.
- مدیریت دسترسی اینباندها برای هر ریسلر؛ همه اینباندها یا اینباندهای اختصاصی.
- حالت تعمیرات برای مسدود کردن موقت کاربران غیرمدیر.
- دکمه پایدار `🏠 داشبورد` برای بازگشت سریع مدیر و ریسلر به منوی اصلی.
- بکاپ دستی و خودکار پایگاه داده SQLite از داخل پنل مدیریت و ارسال فایل دیتابیس برای مدیرها.

## نصب

```bash
git clone <your-repo-url>
cd resell-bot
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

## اجرای ربات با systemd

استفاده از `systemd` باعث می‌شود ربات به‌صورت خودکار در پس‌زمینه اجرا شود، اگر به هر دلیل کرش کرد دوباره راه‌اندازی شود و بعد از ری‌استارت شدن سرور نیز بدون نیاز به اجرای دستی دوباره شروع به کار کند. این روش برای اجرای دائمی ربات روی Ubuntu/Linux پیشنهاد می‌شود.

> در مثال‌های زیر فرض شده پروژه در مسیر `/root/resell-bot` قرار دارد و با کاربر `root` اجرا می‌شود. اگر مسیر پروژه یا کاربر شما متفاوت است، همان مقدارها را در دستورها و فایل سرویس جایگزین کنید.

1. ابتدا وارد پوشه پروژه شوید:

```bash
cd /root/resell-bot
```

2. مطمئن شوید محیط مجازی Python ساخته شده و وابستگی‌های پروژه نصب شده‌اند:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

3. قبل از ساخت سرویس، ربات را یک بار به‌صورت دستی تست کنید تا مطمئن شوید تنظیمات `.env`، اتصال به دیتابیس و اتصال به تلگرام درست هستند:

```bash
python3 run.py
```

اگر ربات بدون خطا اجرا شد، با کلیدهای `Ctrl + C` آن را متوقف کنید و مراحل بعدی را انجام دهید.

4. فایل سرویس `systemd` را بسازید:

```bash
nano /etc/systemd/system/resell-bot.service
```

5. محتوای زیر را داخل فایل قرار دهید، سپس فایل را ذخیره کنید:

```ini
[Unit]
Description=Resell Bot Telegram Service
After=network.target

[Service]
WorkingDirectory=/root/resell-bot
ExecStart=/root/resell-bot/.venv/bin/python /root/resell-bot/run.py
Restart=always
RestartSec=5
User=root
Environment=PYTHONUNBUFFERED=1

[Install]
WantedBy=multi-user.target
```

توضیح کوتاه گزینه‌های مهم:

- `WorkingDirectory` مشخص می‌کند ربات از کدام پوشه اجرا شود؛ این گزینه برای پیدا شدن فایل‌هایی مثل `.env` و دیتابیس بسیار مهم است.
- `ExecStart` مسیر Python داخل محیط مجازی و فایل اجرای ربات را مشخص می‌کند.
- `Restart=always` باعث می‌شود اگر ربات متوقف یا کرش شد، دوباره اجرا شود.
- `RestartSec=5` یعنی قبل از اجرای دوباره، ۵ ثانیه صبر کند.
- `User=root` یعنی سرویس با کاربر `root` اجرا می‌شود. اگر پروژه را با کاربر دیگری نصب کرده‌اید، این مقدار را تغییر دهید.

6. بعد از ساخت یا تغییر فایل سرویس، `systemd` را reload کنید:

```bash
systemctl daemon-reload
```

7. اجرای خودکار ربات بعد از روشن یا ری‌استارت شدن سرور را فعال کنید:

```bash
systemctl enable resell-bot
```

8. ربات را اجرا کنید:

```bash
systemctl start resell-bot
```

9. وضعیت سرویس را بررسی کنید:

```bash
systemctl status resell-bot
```

اگر وضعیت `active (running)` نمایش داده شد، یعنی ربات با موفقیت در پس‌زمینه اجرا شده است.

10. برای دیدن لاگ‌های زنده ربات از دستور زیر استفاده کنید:

```bash
journalctl -u resell-bot -f
```

11. هر زمان بعد از تغییر تنظیمات یا به‌روزرسانی کد نیاز داشتید ربات را دوباره راه‌اندازی کنید:

```bash
systemctl restart resell-bot
```

12. برای توقف کامل ربات از دستور زیر استفاده کنید:

```bash
systemctl stop resell-bot
```

### خطاهای رایج systemd

- اگر فایل `.env` لود نمی‌شود یا ربات تنظیمات را پیدا نمی‌کند، مطمئن شوید مقدار `WorkingDirectory` دقیقاً برابر مسیر پروژه است؛ برای مثال: `/root/resell-bot`.
- اگر مسیر Python اشتباه است یا سرویس با خطای پیدا نشدن Python اجرا نمی‌شود، وجود فایل Python محیط مجازی را بررسی کنید:

```bash
ls /root/resell-bot/.venv/bin/python
```

- اگر وابستگی‌ها نصب نیستند یا خطای `ModuleNotFoundError` می‌بینید، محیط مجازی را فعال کنید و پکیج‌ها را دوباره نصب کنید:

```bash
source /root/resell-bot/.venv/bin/activate
pip install -r requirements.txt
```

- اگر مسیر دیتابیس اشتباه است یا ربات دیتابیس را پیدا نمی‌کند، مقدار `DATABASE_URL` را در فایل `.env` بررسی کنید و مطمئن شوید مسیر دیتابیس با محل واقعی فایل هماهنگ است.
- اگر ربات start نمی‌شود یا سریع متوقف می‌شود، ۱۰۰ خط آخر لاگ سرویس را ببینید:

```bash
journalctl -u resell-bot -n 100 --no-pager
```

بعد از رفع خطا، معمولاً لازم است سرویس را دوباره راه‌اندازی کنید:

```bash
systemctl restart resell-bot
```

## بکاپ و بازیابی SQLite

ربات به‌صورت پیش‌فرض از SQLite استفاده می‌کند. مقدار `DATABASE_URL` مشخص می‌کند فایل دیتابیس کجاست. اگر در `.env` نوشته باشید:

```env
DATABASE_URL=sqlite:///bot.db
```

یا شکل async پیش‌فرض پروژه را داشته باشید:

```env
DATABASE_URL=sqlite+aiosqlite:///bot.db
```

فایل دیتابیس همان فایل زیر است:

```text
bot.db
```

این فایل معمولاً در ریشه پروژه قرار دارد؛ برای مثال اگر پروژه در `/root/resell-bot` باشد، مسیر دیتابیس معمولاً `/root/resell-bot/bot.db` است. اگر `DATABASE_URL` به مسیر دیگری اشاره کند، همان مسیر ملاک است. برای نمونه `sqlite:////absolute/path/bot.db` یعنی فایل دیتابیس در مسیر مطلق `/absolute/path/bot.db` قرار دارد.

### گرفتن بکاپ دستی

قبل از کپی‌کردن فایل دیتابیس، بهتر است ربات را متوقف کنید تا فایل در حین نوشتن کپی نشود:

```bash
systemctl stop reseller-bot
cp /root/resell-bot/bot.db /root/backups/backup-bot-$(date +%F-%H-%M).db
systemctl start reseller-bot
```

نکته‌ها:

- چند بکاپ تاریخ‌دار نگه دارید تا در صورت خراب بودن آخرین بکاپ بتوانید نسخه سالم قبلی را برگردانید.
- فایل `.env` را جداگانه بکاپ بگیرید، چون شامل تنظیمات لازم برای راه‌اندازی است.
- فایل `.env` را هرگز عمومی آپلود نکنید؛ این فایل می‌تواند شامل توکن ربات، اطلاعات ورود مرزبان و شناسه مدیرها باشد.
- بکاپ دیتابیس شامل فایل `.env` یا رازهای محیطی نیست.

### بازیابی بکاپ

برای restore کردن بکاپ SQLite، هنگام اجرای ربات فایل دیتابیس را جایگزین نکنید. مراحل پیشنهادی:

1. ربات را متوقف کنید.
2. فایل `bot.db` فعلی را با فایل بکاپ جایگزین کنید.
3. ربات را دوباره اجرا کنید.

مثال:

```bash
systemctl stop reseller-bot
cp backup-bot.db /root/resell-bot/bot.db
systemctl start reseller-bot
```

اگر مسیر دیتابیس در `DATABASE_URL` متفاوت است، مقصد دستور `cp` را مطابق همان مسیر تغییر دهید.


## ساختار کلی

- `app/handlers`: جریان‌های گفت‌وگوی تلگرام برای مدیر و ریسلر.
- `app/keyboards`: دکمه‌های inline فارسی.
- `app/services`: منطق مرزبان، صورتحساب، گزارش و قالب‌بندی.
- `app/database`: مدل‌ها، نشست پایگاه داده و repositoryها.
- `app/middlewares`: احراز هویت و حالت تعمیرات.
- `app/states`: وضعیت‌های FSM aiogram.

## نمونه تجربه کاربری

- مدیر از پنل مدیریت روی `لیست ریسلرها` می‌زند و نام، شناسه تلگرام، موجودی، قیمت هر گیگابایت، وضعیت، تعداد کاربران و نوع دسترسی اینباند هر ریسلر را می‌بیند.
- ریسلر برای ساخت کاربر، نام کاربری، حجم و روز را وارد می‌کند و با دکمه `✅ تایید` عملیات را نهایی می‌کند.
- ریسلر برای تمدید، ابتدا اطلاعات اکانت را با مالکیت معتبر می‌بیند؛ اگر note برابر `belongs to reseller-name` نباشد پیام `این اکانت متعلق به شما نیست.` نمایش داده می‌شود.


## چک‌لیست تست دستی لیست اکانت‌های ریسلر

- ریسلری که هیچ کاربری نساخته است پیام `هنوز هیچ اکانتی توسط شما ساخته نشده است.` را ببیند.
- ریسلری که کمتر از ۱۰ کاربر ساخته است فقط نام کاربری همان اکانت‌ها را ببیند.
- ریسلری که بیشتر از ۱۰ کاربر ساخته است در صفحه اول فقط ۱۰ نام کاربری جدیدتر را ببیند.
- دکمه `نمایش بیشتر` صفحه بعدی را با ۱۰ نام کاربری بعدی باز کند.
- دکمه `قبلی` صفحه قبل و دکمه `برگشت به داشبورد` داشبورد ریسلر را نمایش دهد.
- ریسلر نتواند کاربران ریسلر دیگر را از طریق callback یا صفحه‌بندی مشاهده کند.
- برای این لیست هیچ فراخوانی به API مرزبان انجام نشود و داده‌ها از جدول محلی `created_users` خوانده شوند.
- نام‌های کاربری از جدیدترین رکورد ساخته‌شده به قدیمی‌ترین مرتب شوند.

## چک‌لیست تست دستی جریان شارژ

- ریسلر درخواست شارژ را با رسید متنی ارسال کند.
- ریسلر درخواست شارژ را با رسید تصویری ارسال کند.
- مدیر روی دکمه `✅ تایید` بزند و درخواست با همان یک کلیک تایید شود.
- موجودی ریسلر دقیقاً یک بار به اندازه مبلغ درخواست افزایش پیدا کند.
- تراکنش موجودی با نوع شارژ تاییدشده ثبت شود.
- ریسلر پیام تایید شارژ را به فارسی دریافت کند.
- مدیر پیام موفقیت تایید شارژ را به فارسی دریافت کند.
- مدیر نتواند همان درخواست را دوباره تایید کند و پیام `این درخواست قبلاً پردازش شده است.` نمایش داده شود.
- مدیر روی دکمه `❌ رد` بزند، دلیل اختیاری را وارد کند یا `رد بدون دلیل` را انتخاب کند و درخواست رد شود.
- با رد درخواست، موجودی ریسلر تغییر نکند.
- callback نامعتبر باعث crash ربات نشود و پیام خطای فارسی نمایش داده شود.

## چک‌لیست تست دستی ساخت کاربر مرزبان

- ساخت یک کاربر جدید با وضعیت `on_hold` را به‌صورت عادی تست کنید.
- ساخت نام کاربری موجود را تست کنید و مطمئن شوید پیام فارسی مناسب نمایش داده می‌شود.
- حالت خطای ۵۰۰ مرزبان پس از ایجاد واقعی کاربر را شبیه‌سازی کنید.
- بررسی کنید ربات بعد از خطای ساخت، `get_user(username)` را فراخوانی می‌کند.
- بررسی کنید اگر یادداشت مالکیت کاربر با `belongs to <reseller_name>` مطابق باشد، عملیات موفق در نظر گرفته می‌شود.
- بررسی کنید موجودی دقیقاً یک‌بار کسر می‌شود.
- بررسی کنید رکورد تکراری در جدول `created_users` ساخته نمی‌شود.
- بررسی کنید تلاش دوباره برای همان نام کاربری باعث کسر دوباره موجودی نمی‌شود.

## چک‌لیست تست دستی ساخت کاربر Marzban

برای اطمینان از هم‌خوانی payload ربات با schema واقعی Marzban، قبل از فعال‌سازی در محیط اصلی این موارد را بررسی کنید:

- یک کاربر نمونه را از پنل Marzban بسازید و ساختار API آن را، مخصوصاً `proxies` و `inbounds`، بررسی کنید.
- همان ساختار proxy/inbound را از طریق ربات برای ساخت کاربر ریسلر استفاده کنید.
- مطمئن شوید `POST /api/user` با وضعیت موفق `200` یا `201` برمی‌گردد.
- مطمئن شوید کاربر ساخته‌شده وضعیت `on_hold` دارد.
- مطمئن شوید برای کاربر `on_hold` مقدار `expire` ارسال نمی‌شود.
- مطمئن شوید `on_hold_expire_duration` با مقدار ثانیه‌ای معتبر ارسال می‌شود.
- مطمئن شوید `note` با قالب `belongs to reseller-name` ذخیره می‌شود.
- مطمئن شوید موجودی ریسلر فقط بعد از موفقیت ساخت کاربر کسر می‌شود.
- مطمئن شوید کاربر بلافاصله بعد از ساخت آنلاین نیست و مقدار `used_traffic` صفر است.
- مطمئن شوید `expire` بعد از ساخت اکانت `on_hold` فعال نشده است.
- مطمئن شوید ربات بعد از ساخت، لینک اشتراک را از پاسخ/اطلاعات کاربر مرزبان استخراج و برای ریسلر ارسال می‌کند.
- مطمئن شوید QR Code لینک اشتراک برای ریسلر ارسال می‌شود و مدیر فقط گزارش عملیات را دریافت می‌کند.

## Multiple Telegram IDs per reseller

Each reseller is one business account with one balance, one price per GB, one inbound permission set, one transaction history, one display name, and one ownership note. A reseller can now link multiple numeric Telegram IDs to that same account through the admin bot UI.

Admin management buttons:

- `👥 اکانت‌های تلگرام ریسلر` shows the Telegram IDs linked to a reseller.
- `➕ افزودن آیدی تلگرام` links another Telegram numeric ID to the selected reseller.
- `➖ حذف آیدی تلگرام` removes a linked Telegram ID, but the bot prevents removing the last ID.
- `⭐ تنظیم به عنوان اصلی` marks one linked ID as the primary Telegram ID.

Telegram ID ownership is globally unique: one Telegram ID cannot be linked to more than one reseller. All linked IDs open the same reseller dashboard and spend from the same shared balance. Created users and renewals remain owned by the reseller display name, so the Marzban note continues to be:

```text
belongs to reseller-name
```

Reports include the reseller display name, the Telegram ID that performed the action, and the primary Telegram ID when available.

## Last app / User-Agent display

When a reseller checks user info during renewal, the bot displays the latest detected client/app User-Agent as:

```text
آخرین برنامه / User-Agent: ...
```

If Marzban has no usable client/app/version information, the bot safely displays:

```text
آخرین برنامه / User-Agent: نامشخص
```

The extractor ignores subscription links, usernames, proxy/inbound names, and IP-only values so they are not confused with a real app/client version.

## چک‌لیست تست دستی قابلیت‌های مستقیم مدیر در مرزبان

برای قابلیت‌های جدید پنل مدیریت، موارد زیر را در یک محیط تست مرزبان بررسی کنید:

- مدیر از مسیر «➕ ساخت کاربر مرزبان» یک کاربر جدید بسازد؛ کاربر باید بدون نیاز به موجودی ریسلر ساخته شود، وضعیت اولیه او `on_hold` باشد و لینک اشتراک/QR در صورت وجود برای مدیر ارسال شود.
- پس از ساخت مستقیم توسط مدیر، موجودی هیچ ریسلری نباید کم شود و رکورد محافظت از ساخت تکراری ریسلرها نباید برای این عملیات ادمین شارژ یا کسر مالی ایجاد کند.
- مدیر از مسیر «♻️ تمدید کاربر مرزبان» یک کاربر متعلق به یک ریسلر را تمدید کند؛ عملیات باید بدون بررسی مالکیت note و بدون کسر موجودی ریسلر انجام شود.
- مدیر یک کاربر مرزبان را که متعلق به هیچ ریسلری نیست تمدید کند؛ عملیات باید بدون نیاز به مالک ریسلری انجام شود.
- ریسلر همچنان نتواند کاربری را که note آن دقیقاً با قالب مالکیت خودش برابر نیست تمدید کند.
- ساخت و تمدید کاربر توسط ریسلر همچنان هزینه را بر اساس قیمت هر گیگابایت از موجودی همان ریسلر کم کند.

## چک‌لیست تست دستی مدیریت کاربر Marzban توسط مدیر

قابلیت‌های غیرفعال‌سازی، فعال‌سازی و حذف فقط از پنل مدیر در دسترس هستند و نباید هیچ موجودی ریسلری را تغییر دهند:

- مدیر از مسیر «⏸ غیرفعال‌سازی کاربر» نام کاربری یک اکانت موجود را وارد کند، صفحه تایید شامل نام کاربری، وضعیت، حجم کل، مصرف‌شده، باقی‌مانده، زمان/انقضا و آخرین برنامه / User-Agent را ببیند و پس از تایید، وضعیت کاربر در Marzban با `PUT /api/user/{username}` و payload `{"status": "disabled"}` غیرفعال شود.
- مدیر از مسیر «▶️ فعال‌سازی کاربر» همان کاربر را دوباره انتخاب کند و پس از تایید، وضعیت کاربر در Marzban با `PUT /api/user/{username}` و payload `{"status": "active"}` فعال شود؛ حجم، انقضا، note، inbounds، proxies و تنظیمات دیگر باید حفظ شوند.
- مدیر از مسیر «🗑 حذف کاربر مرزبان» یک کاربر را انتخاب کند، هشدار دائمی بودن حذف را ببیند، فقط با دکمه «🗑 حذف قطعی» ادامه دهد و کاربر با `DELETE /api/user/{username}` حذف شود.
- مدیر بتواند کاربری را که note آن متعلق به یک ریسلر است غیرفعال، فعال و حذف کند؛ عملیات ادمین نباید بررسی مالکیت ریسلری انجام دهد.
- مدیر بتواند کاربری را که به هیچ ریسلری تعلق ندارد غیرفعال، فعال و حذف کند.
- ریسلر همچنان نتواند اطلاعات یا تمدید کاربری را که note آن متعلق به خودش نیست مدیریت کند و پیام فارسی مالکیت نامعتبر را ببیند.
- قبل و بعد از هر عملیات ادمین، موجودی همه ریسلرهای مرتبط را بررسی کنید؛ غیرفعال‌سازی، فعال‌سازی و حذف ادمین نباید هیچ تراکنش یا کسر موجودی ایجاد کند.
- خطاهای Marzban مثل 404، 401، 500 یا پاسخ نامعتبر را شبیه‌سازی کنید؛ ربات باید پیام امن فارسی نمایش دهد و جزئیات فنی فقط در لاگ ثبت شود.
- لاگ برنامه را بررسی کنید؛ شناسه تلگرام مدیر و نام کاربری کاربر غیرفعال/فعال/حذف‌شده باید ثبت شده باشد.

## چک‌لیست دستی مدیریت وضعیت کاربران مرزبان

برای اعتبارسنجی اصلاحات فعال/غیرفعال/حذف کاربران، این سناریوها را با یک اکانت تست مرزبان و حداقل دو ریسلر اجرا کنید:

- ریسلر از داشبورد گزینه `⏸ غیرفعال‌سازی کاربر` را انتخاب کند، نام کاربری متعلق به خودش را وارد کند، صفحه تایید شامل نام کاربری، وضعیت فعلی، حجم کل، مصرف‌شده، باقی‌مانده، زمان باقی‌مانده و آخرین برنامه / User-Agent را ببیند و پس از تایید کاربر غیرفعال شود.
- ریسلر از داشبورد گزینه `▶️ فعال‌سازی کاربر` را انتخاب کند، نام کاربری متعلق به خودش را وارد کند، همان صفحه تایید را ببیند و پس از تایید کاربر فعال شود.
- ریسلر برای غیرفعال‌سازی کاربر متعلق به ریسلر دیگر تلاش کند و پیام `این اکانت متعلق به شما نیست.` دریافت کند.
- ریسلر برای فعال‌سازی کاربر متعلق به ریسلر دیگر تلاش کند و پیام `این اکانت متعلق به شما نیست.` دریافت کند.
- بررسی کنید هیچ دکمه یا مسیر حذف کاربر در داشبورد ریسلر وجود ندارد و ریسلر نمی‌تواند کاربری را حذف کند.
- مدیر از پنل مدیریت هر کاربر مرزبانی را غیرفعال کند و مالکیت ریسلری بررسی نشود.
- مدیر از پنل مدیریت هر کاربر مرزبانی را فعال کند و مالکیت ریسلری بررسی نشود.
- مدیر کاربری را که متعلق به یک ریسلر است حذف کند؛ حذف فقط در Marzban انجام شود و رکورد موجودی ریسلر تغییر نکند.
- قبل و بعد از حذف مدیر، موجودی همان ریسلر را در پایگاه داده مقایسه کنید و مطمئن شوید دقیقاً یکسان است.
- جدول/گزارش تراکنش‌ها را بررسی کنید و مطمئن شوید برای حذف مدیر هیچ تراکنش، شارژ، بازگشت وجه یا reverse billing entry ایجاد نشده است.
- جدول/گزارش تراکنش‌ها را بررسی کنید و مطمئن شوید برای فعال‌سازی یا غیرفعال‌سازی توسط مدیر یا ریسلر هیچ تراکنش موجودی ایجاد نشده است.
